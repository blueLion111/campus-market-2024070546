# Day3 证据卡：Mock 数据建模与列表渲染

## 一、今日任务总结

Day3 的核心任务是将项目从"静态页面"推进到"数据驱动页面"阶段。通过设计 Mock 数据模型、配置 JSON Server 模拟后端接口、封装 Axios 请求，最终实现四个核心业务页面（二手交易、失物招领、拼单搭子、跑腿委托）的列表渲染。

## 二、Mock 数据设计思路

### 我的设计

在设计 db.json 数据结构时，我围绕校园轻集市的四个核心业务场景进行了建模：

1. **二手交易（trades）**：设计了商品标题、价格、原价、分类、成色、发布人、发布时间、校区、地点、图片、描述、状态、浏览量、收藏量等字段。分类涵盖教材数码、生活用品、交通工具、服饰鞋包等学生常用类别，状态分为 open（在售）和 closed（已售）。

2. **失物招领（lostFounds）**：设计了标题、类型（lost/found）、物品名称、地点、时间、联系人、描述、发布人、校区、图片、状态等字段。类型区分寻物启事和失物招领，符合校园丢失物品的真实场景。

3. **拼单搭子（groupBuys）**：设计了标题、类型、描述、当前人数、目标人数、截止时间、地点、发布人、发布时间、图片、状态等字段。类型包括奶茶拼单、外卖拼单、学习搭子、运动搭子，涵盖了校园生活中常见的拼单和搭子场景。

4. **跑腿委托（errands）**：设计了任务标题、类型、描述、酬劳、取件地点、送达地点、截止时间、发布人、发布时间、校区、图片、状态等字段。类型包括代取快递、代买物品、代办事务，状态分为 open（待接单）、accepted（进行中）、done（已完成）。

在数据量方面，每类数据至少准备了 6-8 条，确保列表展示有足够内容，且数据内容都贴近真实校园场景，避免无关测试数据。

### AI 设计

在设计数据模型时，AI Coding 工具提供了一些参考建议：

**AI 的优点：**
- 提供了非常完整的字段设计，包括创建时间、更新时间、软删除标记等后端常用字段
- 建议增加用户表、评论表、收藏表、订单表等关联数据结构
- 建议使用更规范的数据库命名方式（如 snake_case）
- 提出了更多状态细分，如待审核、已下架等

**AI 的问题：**
- 设计过于复杂，Day3 阶段不需要后端级别的完整数据模型
- 引入了太多关联表，增加了理解和维护成本
- 部分字段（如 updatedAt、deletedAt）在纯前端 Mock 阶段完全用不上
- 没有充分考虑校园场景的特殊性，部分分类不够贴近学生生活

### 最终调整

经过人工审查和调整，我最终采用的方案是：

1. **保留**：四类核心业务数据结构（trades、lostFounds、groupBuys、errands），这是页面展示的基础
2. **修改**：将 AI 建议的复杂关联表简化为单表结构，所有必要信息直接存储在业务对象中
3. **删除**：移除了用户表、评论表、收藏表、订单表等 Day3 不需要的数据表
4. **新增**：根据校园场景特点，增加了校区（campus）、成色（condition）等学生关注的字段
5. **规范**：统一使用驼峰命名（camelCase），与前端 Vue 项目的代码风格保持一致好

这样的设计既满足了页面列表渲染的需要，又保持了数据结构的简洁性，避免了过度设计。

## 三、JSON Server 与接口配置

### JSON Server 启动配置

在 `package.json` 中配置了 mock 启动命令：

```json
"mock": "json-server --watch db.json --port 3001"
```

选择 3001 端口是为了避免与常见的 3000 端口冲突，同时与前端 5173 端口区分开。JSON Server 会自动根据 db.json 生成 RESTful API，例如：
- GET /trades - 获取二手商品列表
- GET /trades/1 - 获取 ID 为 1 的商品详情
- GET /trades?category=textbook - 按分类筛选商品

### Axios 封装

在 `src/api/http.ts` 中进行了基础的 Axios 封装：

- baseURL 设置为 `http://localhost:3001`，指向 JSON Server
- timeout 设置为 5000ms，防止请求超时过长

Day3 阶段刻意保持了封装的简洁性，没有添加复杂的请求/响应拦截器、JWT Token 处理、错误码统一处理等，因为这些是后续阶段的任务。

### 业务 API 模块化

将接口请求按业务模块拆分到不同文件中：
- `src/api/trade.ts` - 二手交易相关接口
- `src/api/lostFound.ts` - 失物招领相关接口
- `src/api/groupBuy.ts` - 拼单搭子相关接口
- `src/api/errand.ts` - 跑腿委托相关接口

每个模块都导出了对应的 TypeScript 类型定义和请求函数，这样做的好处是：
- 页面组件不需要关心具体的接口路径和实现细节
- 类型定义集中管理，便于维护
- 后续接入真实后端时，只需修改 API 层即可，不需要改动页面代码

## 四、页面列表渲染实现

### 数据驱动改造

四个核心页面都从"静态数据写死在组件中"改造为"通过 API 请求获取数据并驱动渲染"：

1. 在 `onMounted` 生命周期钩子中调用 API 请求数据
2. 使用 `ref` 存储响应式数据
3. 使用 `computed` 进行分类筛选
4. 模板中通过 `v-for` 渲染列表

### 通用组件封装

- **ItemCard.vue**：通用列表卡片组件，支持图片、标题、价格、地点、发布人等信息展示，可在多个页面复用
- **EmptyState.vue**：空状态组件，当列表数据为空时展示友好提示

### 状态考虑

在实现过程中，我考虑了以下几种页面状态：
- **加载中**：loading 状态（Day3 简单处理，后续可完善加载动画）
- **请求成功**：正常渲染列表数据
- **数据为空**：展示 EmptyState 空状态组件
- **请求失败**：控制台输出错误信息（后续可增加用户友好的错误提示）

## 五、AI 协作与人工判断

### AI 使用场景

今天在以下环节使用了 AI Coding 工具辅助：
1. Mock 数据结构设计参考
2. db.json 数据内容生成
3. Axios 封装代码参考
4. API 模块类型定义生成

### 人工判断过程

对于 AI 生成的内容，我进行了逐项审查：

1. **数据结构审查**：AI 倾向于生成大而全的数据库结构，但我根据 Day3 的实际需求进行了裁剪，只保留页面展示必需的字段
2. **数据内容审查**：AI 生成的数据有些过于通用，我调整为更贴近校园场景的内容，如增加校区信息、调整分类名称等
3. **代码质量审查**：AI 生成的 API 封装有时会过度设计，我坚持 Day3 简洁优先的原则，只做基础封装
4. **命名规范审查**：统一使用驼峰命名，确保前后端字段命名风格一致

## 六、今日收获

通过 Day3 的学习和实践，我深入理解了：

1. **Mock 数据的重要性**：Mock 数据不是随便编造的测试数据，而是前端开发过程中理解业务、验证页面、模拟后端接口的重要工具
2. **数据驱动的思想**：页面展示不应该是写死的文本，而是先分析业务对象，设计数据结构，再通过接口请求将数据渲染到页面
3. **接口分层的好处**：将 API 请求从页面组件中分离出来，使代码结构更清晰，维护更方便
4. **AI 协作的正确姿势**：AI 可以提供参考和建议，但最终的设计决策需要人工判断，要根据项目实际阶段和需求进行取舍

Day3 是从静态页面到动态数据驱动的关键一步，合理的数据结构设计将为后续 Day4 的发布表单、Day5 的状态管理等任务打下坚实基础。
