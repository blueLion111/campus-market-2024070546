# Day5 证据卡——状态管理与用户中心

## 一、今日任务概览

Day5 的核心主题是 **Pinia 状态管理** 和 **用户中心** 功能实现。在 Day4 完成发布表单、表单校验和数据新增的基础上，本日引入了前端状态管理机制，使用 Pinia 管理当前用户信息和收藏状态，实现了多页面之间的数据共享。

本日的目标不是实现真实的登录注册系统，而是理解「当项目从单个页面扩展为多个页面协同时，很多数据不能只存放在某个页面组件内部，而需要通过统一的状态管理方式进行维护和共享」这一核心概念。

---

## 二、Pinia Store 设计说明

### 2.1 Store 划分与职责

本日共设计了两个 Pinia Store，均位于 `src/stores/` 目录下：

| Store 文件 | 主要职责 | 核心状态 |
| --- | --- | --- |
| `user.ts` | 管理当前用户信息 | currentUser、isLoggedIn、nickname、avatar、college 等 |
| `favorite.ts` | 管理用户收藏状态 | favorites 列表、favoriteCount、isFavorite、toggleFavorite |

采用 Composition API 风格的 `defineStore` 写法，使用 `ref` 定义 state、`computed` 定义 getters、普通函数定义 actions，符合 Vue 3 官方推荐的写法。

### 2.2 user Store 设计

`useUserStore` 的设计要点：

- **状态**：`currentUser`（User 类型，可为 null）、`isLoggedIn`（布尔值）
- **Getters**：`nickname`、`avatar`、`college`、`grade`、`major`、`campus` 等计算属性，均通过 `currentUser` 派生，避免在模板中写长链访问
- **Actions**：
  - `setUser(user)`：设置当前用户，同时标记为已登录
  - `clearUser()`：清除用户信息，退出登录
  - `updateProfile(data)`：部分更新用户资料

用户状态在 `App.vue` 的 `onMounted` 钩子中通过调用 `getUserById(1)` 从 Mock 接口获取并初始化，若接口失败则保留默认模拟用户（此处为了简化 Day5 流程，直接使用 id=1 的用户作为当前登录用户）。

### 2.3 favorite Store 设计

`useFavoriteStore` 的设计要点：

- **状态**：`favorites`（FavoriteItem 数组）
- **Getters**：
  - `favoriteCount`：收藏总数
  - `isFavorite(itemId, itemType)`：判断某个物品是否已收藏，使用 itemId + itemType 双重判断以支持不同业务类型
- **Actions**：
  - `addFavorite(item)`：添加收藏，自动记录收藏时间
  - `removeFavorite(itemId, itemType)`：移除收藏
  - `toggleFavorite(item)`：切换收藏状态，返回是否被收藏（布尔值）
  - `clearFavorites()`：清空所有收藏

收藏状态目前保存在前端内存中，刷新页面后会重新初始化。这是 Day5 阶段的设计选择——重点在于理解 Pinia 的状态共享机制，持久化可作为后续优化方向。

## 三、状态边界说明

Pinia 状态管理中一个关键问题是：**哪些数据应该放入 Store，哪些不应该？** 本日遵循以下原则进行了状态边界划分：

### 3.1 放入 Store 的数据

| 数据 | 放入原因 |
| --- | --- |
| 当前用户信息（昵称、头像、学院等） | 导航栏、发布页、个人中心、详情页等多个页面/组件都需要使用 |
| 收藏列表 | 列表页（用于显示收藏状态）、个人中心（用于展示我的收藏）、导航栏（收藏计数）都需要访问 |

### 3.2 不放入 Store 的数据

| 数据 | 不放入原因 |
| --- | --- |
| 各业务列表数据（二手商品、失物招领等） | 仅在对应列表页面使用，每个页面有独立的筛选和分页逻辑，放入 Store 反而增加复杂度 |
| 表单输入内容和校验信息 | 仅属于当前发布表单页面，与其他页面无关 |
| 页面临时 loading 状态 | 各页面独立维护即可 |
| 单个商品的详情数据 | 只在详情页使用，通过路由参数获取并请求接口即可 |

**状态边界意识总结**：Store 不是越大越好，而是存放「真正需要跨组件/跨页面共享」的数据。把所有数据都塞进 Store 会导致状态膨胀、难以维护；只有明确哪些数据是共享的，哪些是局部的，才能设计出清晰合理的状态结构。

## 四、跨页面状态使用场景

### 4.1 导航栏（AppHeader.vue）

导航栏右侧显示当前登录用户的头像和昵称，数据来源于 `useUserStore`。当用户状态变化时（如未来实现登录/登出），导航栏会自动更新。

### 4.2 发布页面（PublishView.vue）

发布数据时，`publisher` 和 `publisherAvatar` 字段不再写死，而是从 `userStore.nickname` 和 `userStore.avatar` 读取，保证发布数据与当前登录用户关联。

### 4.3 列表页面（TradeView.vue）

二手交易列表的每个商品卡片右上角添加了收藏按钮（⭐/☆），点击可切换收藏状态。收藏状态通过 `favoriteStore.isFavorite(product.id, 'trade')` 判断，点击时调用 `favoriteStore.toggleFavorite()`。

使用 `e.stopPropagation()` 防止收藏按钮点击事件冒泡到卡片的点击跳转事件上，避免点击收藏同时跳转到详情页。

### 4.4 个人中心页面（UserCenterView.vue）

个人中心是状态使用最集中的页面：

- **用户资料区**：头像、昵称、学院、专业、年级、校区、信用分等全部从 `userStore.currentUser` 读取
- **统计卡片**：我的收藏数量直接使用 `favoriteStore.favoriteCount`，响应式更新
- **我的收藏区块**：展示 `favoriteStore.favorites` 中的前 4 条收藏，支持取消收藏操作
- **最近动态**：结合发布数据和收藏数据生成，收藏变化时通过 `watch` 监听 `favoriteCount` 并更新动态列表

## 五、AI 协作记录

### 5.1 AI 帮助生成的内容

1. **Pinia Store 初始结构**：AI 辅助生成了 `useUserStore` 和 `useFavoriteStore` 的基本框架，包括 state、getters、actions 的组织方式，以及 TypeScript 类型定义（User 接口、FavoriteItem 接口）。

2. **收藏功能基础逻辑**：AI 给出了 `toggleFavorite` 的实现思路——通过 `some()` 方法判断是否已收藏，再根据结果调用 `addFavorite` 或 `removeFavorite`。

3. **个人中心收藏区块样式**：AI 帮助生成了收藏卡片的 CSS 样式（卡片布局、悬停效果、取消按钮样式等）。

### 5.2 人工调整和审查的内容

| AI 生成了「浏览历史」「最近访客」等无关功能 | 校园轻集市 Day5 只要求收藏和用户中心，多余功能属于过度设计 | 全部删除，只保留实验手册要求的核心功能 |
| 个人中心生成了复杂的仪表盘图表 | Day5 重点是状态管理，不是数据可视化，图表属于无意义占位 | 删除图表，改为简洁的统计卡片展示 |

#### 三、代码简洁性审查（剔除未要求的复杂权限与接口）

| AI 生成的冗余内容 | 人工审查结论 | 处理方式 |
| --- | --- | --- |
| 完整登录/注册页面（含表单校验、验证码） | Day5 不要求真实登录系统，仅需模拟用户状态 | 全部删除，仅保留 `userStore.login()` 作为模拟登录入口 |
| JWT Token 存储与刷新机制 | 超出 Day5 范围，属于后端认证范畴 | 删除 Token 相关代码 |
| 路由权限守卫（未登录跳转到登录页） | Day5 是模拟用户，不需要权限拦截 | 删除路由守卫 |
| 复杂的权限系统（角色、菜单、按钮级权限） | 校园轻集市没有多角色权限需求 | 全部删除 |
| 收藏持久化（localStorage + 后端双写） | Day5 重点是理解 Pinia 状态共享，持久化是进阶内容 | 删除 localStorage 相关代码，仅保留后端同步（因为 JSON Server 已有数据） |
| Store 中多余的异步 action（如 fetchUser、fetchFavorites） | 数据获取应该在页面组件中发起，Store 只负责状态管理 | 移除异步请求，Store 只保留纯状态操作 |

#### 四、其他人工调整

1. **Store 语义重构**：AI 生成的 Store 中 state/getter/action 混在一起，命名冗长（如 `addFavorite`、`isFavorite`）。人工按照「State 名词、Getter 名词/判断、Action 动词」的原则重命名，明确语义边界，代码结构更清晰。

2. **收藏按钮事件冒泡**：AI 生成的收藏按钮直接放在卡片内部，点击会触发卡片跳转。人工添加 `e.stopPropagation()` 修复交互问题。

3. **状态边界把控**：AI 建议把所有业务列表数据都放入 Store。人工审查后认为违反状态边界原则——只有跨页面共享的数据（用户信息、收藏列表）才应该放 Store，各页面的列表数据属于页面临时数据，保留在组件内部。

4. **类型定义统一**：AI 生成的 `FavoriteItem` 字段命名不一致，人工统一为 `itemId` / `itemType`，与后端 API 的 `Favorite` 接口对齐，确保数据转换时不出错。

## 七、收获与体会

通过 Day5 的实训，我对 **Pinia** 和前端 **状态管理** 有了更深入的理解：

1. **状态管理的价值在于解决共享问题**：之前用户信息写死在各个页面里，现在通过 Store 统一管理，修改一处全局生效，避免了数据不一致的问题。

2. **Store 设计需要边界意识**：不是所有数据都适合放进 Store。理解「哪些数据是共享的，哪些是局部的」比掌握 Pinia API 更重要。

3. **响应式是 Pinia 的核心优势**：在列表页收藏一个商品，个人中心的收藏数量和收藏列表会自动更新，不需要手动通知，这就是响应式状态管理的魅力。

4. **用户中心是状态管理的集中体现**：**用户中心** 页面汇聚了来自多个 Store 的数据（用户资料 + 收藏列表），非常直观地展示了状态管理在实际项目中的应用价值。

5. **AI 是工具，判断在人**：AI 可以快速生成代码框架，但状态边界的把控、业务逻辑的合理性、交互细节的处理，仍然需要人工审查和决策。
