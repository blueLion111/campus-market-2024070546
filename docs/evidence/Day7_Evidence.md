## 六、AI 协作复盘

### 6.1 AI 帮助最大的阶段

1. **页面骨架搭建（Day2）**：AI 快速生成 12 个页面的基础结构和路由配置，节省了大量重复劳动。
2. **Mock 数据设计（Day3）**：AI 辅助设计数据结构和生成示例数据，保证了数据的完整性和多样性。
3. **组件样式优化（Day5-6）**：AI 帮助将彩色 PPT 风格转换为极简主义风格，统一各页面样式。
4. **状态组件封装（Day6）**：AI 快速生成 LoadingState、ErrorState、SearchBar 等组件的基础代码。
5. **证据卡文档编写（Day6-7）**：AI 辅助整理开发过程和问题记录，形成结构化的证据卡。

### 6.2 AI 生成内容中的问题

1. **过度设计**：AI 总是倾向于给出"最完美"的方案，比如建议引入 VueUse、VeeValidate 等额外库，增加了项目复杂度。
2. **忽略上下文**：有时 AI 会忘记项目的实际约束（比如这是教学实训项目，不需要生产级别的复杂度）。
3. **样式偏差**：AI 生成的样式有时会偏离极简主义要求，需要反复调整和人工筛选。
4. **类型不严谨**：AI 生成的 TypeScript 代码经常使用 any 类型，需要人工修正。
5. **虚构功能**：AI 有时会"脑补"一些不存在的功能，比如在 README 中写了登录注册、支付等未实现的功能。

### 6.3 人工判断与修改

1. **功能筛选**：所有 AI 建议的功能都经过人工评估，只保留符合 Day1-Day7 实训范围的内容。
2. **风格把控**：极简主义风格主要由人工定义设计系统，AI 负责将其应用到各个页面。
3. **代码审查**：AI 生成的每一段代码都经过人工检查，确保逻辑正确、类型安全、命名规范。
4. **文档审核**：AI 辅助生成的文档都经过人工核对，确保描述的功能都是真实实现的，没有夸大。
5. **问题定位**：遇到 bug 时，AI 提供排查思路，但最终定位和修复由人工完成。

### 6.4 必须人工理解的内容

1. **业务逻辑**：四类业务（二手、失物、拼单、跑腿）的数据结构和流程差异，必须人工理解和设计。
2. **状态管理设计**：Pinia store 的结构划分、getter/action 的语义，需要人工规划。
3. **设计系统**：极简主义的设计理念、CSS 变量体系、组件风格统一，需要人工把控。
4. **问题调试**：遇到运行时错误，AI 只能给出猜测，真正的调试还得靠人工看报错、看代码。
5. **项目整体把控**：哪些功能先做、哪些后做、做到什么程度，需要人工根据时间和目标决策。

### 6.5 未来使用 AI 的注意事项

1. **明确边界**：在让 AI 写代码之前，先明确告诉它项目的约束和范围，避免它"自由发挥"。
2. **小块迭代**：不要让 AI 一次性重写整个文件，而是一小段一小段地修改，便于控制和验证。
3. **交叉验证**：AI 给出的方案，不要直接用，先自己想想有没有道理，再测试验证。
4. **保留判断**：AI 说的不一定对，特别是涉及业务逻辑和架构设计的部分，一定要有自己的判断。
5. **重在效率**：AI 最大的价值是提高效率（写样板代码、生成文档、排查思路），而不是代替思考。

## 八、项目不足与改进方向

### 8.1 当前不足

1. **后端模拟**：目前使用 JSON Server 作为 Mock 数据，不是真实后端服务，没有真正的用户认证、权限控制等。
2. **用户系统简化**：用户状态是硬编码的模拟数据，没有真实的登录注册流程。
3. **收藏持久化**：收藏数据存在后端，但刷新页面后需要重新拉取，状态管理可以更完善。
4. **搜索功能基础**：目前只有简单的关键词过滤，没有高级筛选、排序、分页等功能。
5. **TypeScript 类型**：部分地方使用了 any 类型，类型安全可以进一步提升。
6. **移动端适配**：目前主要针对桌面端设计，移动端响应式适配尚未完成。
7. **图片上传**：发布信息时图片使用 URL 输入，没有真正的图片上传功能。
8. **实时通信**：消息中心是静态数据，没有实时消息推送能力。

### 8.2 后续改进方向

1. **接入真实后端**：使用 Node.js / Python / Java 等开发真实后端服务，替换 JSON Server。
2. **完善用户系统**：实现注册、登录、个人信息修改、头像上传等功能。
3. **增强搜索筛选**：增加多条件组合筛选、排序、分页、价格区间等高级功能。
4. **类型安全提升**：逐步替换所有 any 类型，完善接口定义。
5. **响应式适配**：适配移动端、平板等不同屏幕尺寸。
6. **性能优化**：图片懒加载、路由懒加载、虚拟列表等。
7. **单元测试**：为核心组件和 store 添加单元测试。
8. **国际化**：支持中英文切换。

### 9.1 关于前端开发

以前觉得前端就是写页面、调样式，现在才明白一个完整的前端项目包含很多方面：

- **业务理解是前提**：不理解业务场景，就设计不好数据结构和页面流程。
- **架构设计很重要**：组件怎么划分、状态怎么管理、路由怎么组织，这些决定了项目的可维护性。
- **用户体验是关键**：功能做出来只是第一步，加载快不快、出错有没有提示、操作顺不顺手，这些决定了用户愿不愿意用。
- **代码质量是底线**：类型安全、命名规范、注释清晰、结构合理，这些决定了项目能不能持续迭代。

### 9.2 关于 AI 协作

AI Coding 工具确实大大提高了开发效率，但它不是万能的：

- **AI 是加速器，不是发动机**：方向、设计、决策这些核心的东西还是得靠人。AI 只是帮你把想法更快地变成代码。
- **会提问比会写代码更重要**：给 AI 的需求越清晰、越具体，它生成的代码质量就越高。模糊的需求只会得到模糊的答案。
- **审查能力决定了 AI 的价值**：如果看不懂 AI 写的代码、判断不了对错，那 AI 反而可能帮倒忙。
- **要会用工具，更要懂原理**：知道为什么这么写，比知道怎么写更重要。不然出了问题都不知道怎么查。

### 9.3 关于工程实践

这次实训让我体会到了软件工程的"工程"二字：

- **循序渐进**：从 Day1 的业务梳理到 Day7 的综合验收，每天都在前一天的基础上推进，节奏很稳。
- **过程比结果重要**：7 天的证据卡记录了完整的开发过程，包括遇到的问题、解决的思路、AI 的协作，这些比最终的页面更有价值。
- **稳定优先于花哨**：Day6 没有堆新功能，而是优化已有功能的体验，这个思路很对。一个稳定可用的项目，比一个功能多但 bug 满天飞的项目强得多。
- **文档是项目的一部分**：以前觉得写文档是浪费时间，现在发现清晰的 README 和证据卡，能让别人（甚至是几天后的自己）快速理解项目。

7 天的实训结束了，但前端学习的路还很长。希望以后能把这 7 天学到的东西用到更多项目中去。
